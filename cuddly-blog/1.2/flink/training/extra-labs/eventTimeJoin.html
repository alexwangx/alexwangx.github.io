丰富的 Joins

介绍

本练习的目的是探索通过将客户信息作为客户更新流到达而丰富金融交易流的技术。 关于富集 join的思考方式有多种，本练习的目的是探讨有关此主题的一些变体。

我们已经为 Trade 流和 Customer 流实现了源。 这些来源是简单的模拟，仅提供有关一个客户以及该客户交易的事件。 这两个流可以在它们共有的 customerId 字段上连接。

数据类型

Trade 记录包含：

timestamp      : long    // timestamp (milliseconds since the epoch)
customerId     : long    // a unique id for each event
tradeInfo      : String  // payload

Customer 记录包含：

timestamp      : long    // timestamp (milliseconds since the epoch)
customerId     : long    // a unique id for each event
customerInfo   : String  // payload

join 的结果将是一个 EnrichedTrade ：

trade          : Trade
customer       : Customer

  • Trade.java

  • ^EnrichedTrade.java

处理时间join

这是实施 join 的最简单方法，适用于欺诈检测之类的用例，在这些案例中，精确，确定的结果不是必须的，而保持较低的延迟非常有价值。 这里的总体思路是，利用可获得的任何客户信息立即 join 每笔交易。

您可以在 ProcessingTimeJoinExercise 类中找到这种连接的有效实现。

练习

  1. 为此实现编写一些测试。

  2. 思考的问题： ProcessingTimeJoinExercise 实现了类似于外部联接的操作（可能存在没有匹配的 Customer 的 Trade ，在这种情况下，此实现将产生 (Trade, null) 作为结果）。 在某些情况下，如果有匹配的 Customer 到达，那么值得等待一段时间吗？ 您将如何实施呢？

Event Time Join

处理 Time Join 易于实现，并且可以以最小的延迟生成，但不适用于某些应用程序。 如果您需要可重现的确定性结果，则应使用交易时可知的最新客户信息来进行扩展。

以下是进行活动时间加入所涉及的内容：

  1. 将每个交易以键控状态存储，直到当前水印达到交易的时间戳为止。

  2. 存储所有可能相关的客户记录（而不是像处理时加入那样仅存储最新记录）。

  3. 在适当的时候，将适当版本的客户信息加入每个交易中。

  4. 最终清除不再需要的客户数据。

您将在 EventTimeJoinExercise 类中找到这种连接的有效实现。

测验

您可以在以下位置找到此实现的测试

com.ververica.flinktraining.exercises.datastream_java.process.EventTimeJoinTest

练习

  1. 扩展此实现以处理后期交易。 这将要求对过期的客户记录不那么积极。

  2. 实现一个低延迟变体，该变体立即与可用的客户数据（如果有的话）结合在一起，然后在客户更新到达时发出更新的结果。

  3. 额外的：扩展测试以涵盖此新功能。
