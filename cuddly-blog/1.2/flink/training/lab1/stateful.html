有状态转换

为什么 Flink 要管理状态

你的应用程序当然能够使用状态, 而无需Flink参与管理状态-但是Flink为它管理的状态提供了一些引人注目的功能:

  • local: Flink 状态保持在处理它的节点本地, 并且可以以内存速度访问.

  • 持久: Flink 状态可以自动 checkpoint 并恢复

  • 纵向扩展: Flink 状态可以保留在嵌入式 RocksDB 实例中, 该实例可以通过添加更多本地磁盘来扩展

  • 水平扩展: 随着群集的增长和收缩, Flink 状态可以重新分配

  • 可查询: 可以通过 REST API 查询 Flink 状态

在本课程中, 您将学习如何使用Flink的API管理 keyed 状态.

Rich Functions

至此, 我们已经看到了 Flink 的几个功能接口, 包括 FilterFunction , MapFunction 和 FlatMapFunction . 这些都是“单一抽象方法”模式的示例.

对于每个接口, Flink还提供了一个所谓的 “丰富” 变体, 例如 RichFlatMapFunction , 它具有一些其他方法, 包括:

  • open(Configuration c)

  • close()

  • getRuntimeContext()

在 operator 初始化期间, open() 被调用一次. 例如, 这是加载一些静态数据或打开与外部服务的连接的机会.

getRuntimeContext() 提供对上下文的访问, 但是最值得注意的是, 它是您如何创建和访问由 Flink 管理的状态的方法.

一个 Keyed State 的例子

在此示例中, 我们具有由 Tuple2<String, Double> 事件组成的传感器读数流, 这些事件指定每个传感器读数的传感器ID和值. 我们的目标是对来自每个传感器的数据进行平滑处理, 我​​们将使用称为 Smoother 的 RichMapFunction 进行处理.

DataStream<Tuple2<String, Double>> input = …
DataStream<Tuple2<String, Double>> smoothed = input.keyBy(0).map(new Smoother());

为此, 我们的 Smoother 将需要以某种方式记录每个传感器的最新数据, 这将使用 Flink 的 keyed state 接口来完成.

当您使用像这样的 keyed 流时, Flink 将为所管理的每个状态项维护一个 key/value 存储.

Flink 支持几种不同类型的 keyed state, 但是在此示例中, 我们将使用最简单的一种, 即 ValueState . 这意味着Flink将为每个key 储一个对象, 在这种情况下, 将存储为 MovingAverage 类型的对象. 出于性能原因, Flink 为特定类型(包括 ListState 和 MapState )提供了特殊支持.

Smoother 类有两种方法: open() 和 map(). 在 open 方法中, 我们通过定义 ValueStateDescriptor<MovingAverage> 来建立对托管状态的使用. 构造函数的参数为此 keyed state ("moving average") 指定一个名称, 并提供可用于序列化这些对象的信息(在本例中为类MovingAverage.class).

public static class Smoother extends RichMapFunction<Tuple2<String, Double>, Tuple2<String, Double>> {
  private ValueState<MovingAverage> averageState;

  @Override
  public void open (Configuration conf) {
    ValueStateDescriptor<MovingAverage> descriptor =
      new ValueStateDescriptor<>("moving average", MovingAverage.class);
    averageState = getRuntimeContext().getState(descriptor);
  }

  @Override
  public Tuple2<String, Double> map (Tuple2<String, Double> item) throws Exception {
    // access the state for this key
    MovingAverage average = averageState.value();

    // create a new MovingAverage (with window size 2) if none exists for this key
    if (average == null) average = new MovingAverage(2);

    // add this event to the moving average
    average.add(item.f1);
    averageState.update(average);

    // return the smoothed result
    return new Tuple2(item.f0, average.getAverage());
  }
}

Smoother 中的 map 方法负责使用 MovingAverage 平滑每个事件. 每次 map 处理一个 event 时, 该 event 都与特定的键(即特定的传感器)相关联, 并且我们的 ValueState 对象 (averageState) 上的方法隐式地作用于与该 key 绑定的状态下进行操作. 因此, 换句话说, 调用 averageState.value() 返回相应传感器的当前`MovingAverage` 对象, 因此, 当我们调用 average.add(item.f1) 时, 我们将此事件添加到同一键的先前事件中(即 同一传感器).

清理状态

上面的示例存在一个潜在的问题: 如果 key 空间不受限制, 将会发生什么？Flink 为每个使用的不同的 key 在某处存储一个 MovingAverage 实例. 如果传感器数量有限, 那就很好了, 但是在 keys 无限制增长的应用中, 有必要清除不再需要的 key 的状态. 这是通过在状态对象上调用 clear() 来完成的, 如下所示:

averageState.clear()

例如, 您可能要在指定的 key 不活动一段时间之后执行此操作. 再我们学习关于 ProcessFunction 的课程时 event-driven applications, 将会看到如何使用 Timers 去实现.

1.6版中的 Flink 还添加了 “状态生存时间”（TTL）特性. 您可以使用它来指定何时希望自动清除陈旧 key 的状态.

Non-keyed State

也可以在 non-keyed 上下文中使用托管状态. 有时将其称为 operator 状态. 所涉及的接口有些不同, 并且由于用户定义的函数通常需要 non-keyed 状态, 因此在此不再赘述.
