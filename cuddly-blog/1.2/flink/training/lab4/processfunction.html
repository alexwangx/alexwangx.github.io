ProcessFunction

ProcessFunction 将事件处理与计时器和状态相结合，使其成为流处理应用程序的强大构建块。 这是使用Flink创建事件驱动的应用程序的基础。 它与 RichFlatMap 非常相似，但是增加了计时器。

在GitHub的 CarEventSort中，有一个使用 ProcessFunction 对来自联网汽车的事件流进行排序的完整示例，这是您可能已经克隆的训练练习库的一部分。 让我们详细研究一下该应用程序的工作方式。

events.keyBy((ConnectedCarEvent event) -> event.carId)
	.process(new SortFunction())

在此代码段中，我们看到一个称为SortFunction的ProcessFunction被应用于键控流。 这意味着我们将（按时间戳记）分别对每辆车的事件进行排序，而不是试图对整个流进行总体的全局排序-这是无法并行完成的。

SortFunction：

public static class SortFunction extends KeyedProcessFunction<String, ConnectedCarEvent, ConnectedCarEvent> {
	/* we'll use a PriorityQueue to buffer not-yet-fully-sorted events */
	private ValueState<PriorityQueue<ConnectedCarEvent>> queueState = null;

	@Override
	public void open(Configuration config) {
	    /* set up the state we want to use */
		...
	}

	@Override
	public void processElement(ConnectedCarEvent event, Context context, Collector<ConnectedCarEvent> out) throws Exception {
		/* add/sort this event into the queue */
		...

		/* set an event-time timer for when the stream is complete up to the event-time of this event */
		...
	}

	@Override
	public void onTimer(long timestamp, OnTimerContext context, Collector<ConnectedCarEvent> out) throws Exception {
	    /* release the items at the head of the queue that are now ready, based on the CurrentWatermark */
		...
	}
}

注意事项:

  • 有几种类型的ProcessFunctions –这是一个KeyedProcessFunction，但也有CoProcessFunctions，BroadcastProcessFunctions等。

  • KeyedProcessFunction是RichFunction的一种。作为RichFunction，它可以访问使用托管键控状态所需的open和getRuntimeContext方法。

  • 有两个回调要实现：processElement和onTimer。 每个传入事件都会调用processElement； 计时器触发时调用onTimer。 这些可以是事件时间或处理时间计时器。 这两个回调都提供了一个上下文对象，该对象可用于与TimerService交互（以及其他功能）。 两个回调也都传递了一个收集器，该收集器可用于发出结果。

open()

@Override
public void open(Configuration config) {
	ValueStateDescriptor<PriorityQueue<ConnectedCarEvent>> descriptor = new ValueStateDescriptor<>(
		"sorted-events", TypeInformation.of(new TypeHint<PriorityQueue<ConnectedCarEvent>>() {})
	);
	queueState = getRuntimeContext().getState(descriptor);
}

到目前为止，我们在创建 ValueStateDescriptor 时使用了 TaxiRide.class 之类的类型来提供类型信息。在涉及泛型的情况下，Flink需要更多信息。

processElement()

@Override
public void processElement(ConnectedCarEvent event, Context context, Collector<ConnectedCarEvent> out) throws Exception {
	TimerService timerService = context.timerService();

	if (context.timestamp() > timerService.currentWatermark()) {
		PriorityQueue<ConnectedCarEvent> queue = queueState.value();
		if (queue == null) {
			queue = new PriorityQueue<>(10);
		}
		queue.add(event);
		queueState.update(queue);
		timerService.registerEventTimeTimer(event.timestamp);
	}
}

注意事项：

  • 延迟的事件会怎样？水印后面（即较晚）的事件将被丢弃。如果您想做得更好，请考虑使用侧面输出，下一节 将对此进行说明。

  • 我们正在为 event.timestamp 设置一个事件时间计时器。这实际上是一种非常常见的模式。您可以这样想：“请解决所有早于此事件的无序影响事件时，请叫醒我。”

onTimer()

当乱序不再成为问题的时候到了，我们可以释放队列中当前水印之前的所有事件。 它们已经正确排序，可以使用了，假设您可以信任水印，那么现在应该已经到了更早的时间。

可能有多个事件要发布，因为可能有多个具有相同时间戳的事件。 Flink对计时器进行重复数据删除-它只会为给定的时间戳和密钥创建一个计时器-因此我们不能保证计时器和事件之间是一对一的关系。

@Override
public void onTimer(long timestamp, OnTimerContext context, Collector<ConnectedCarEvent> out) throws Exception {
	PriorityQueue<ConnectedCarEvent> queue = queueState.value();
	Long watermark = context.timerService().currentWatermark();
	ConnectedCarEvent head = queue.peek();
	while (head != null && head.timestamp <= watermark) {
		out.collect(head);
		queue.remove(head);
		head = queue.peek();
	}
}

性能考量

您应该意识到，如果您使用的是RocksDB状态后端，这不是实现排序的好方法。 使用RocksDB，您的工作状态以序列化字节的形式存在于磁盘上。 这意味着每个状态访问都需要反序列化，在这种情况下，这意味着反序列化整个 PriorityQueue 。 更新同样昂贵。

Flink提供针对RocksDB优化的 MapState 和 ListState 类型。 在可能的情况下，应使用这些值代替持有某种集合的 ValueState 对象。 RocksDB状态后端可以不通过 ser/de 附加到 ListState ，对于 MapState ，每个 key/value 对都是一个单独的RocksDB对象，因此可以高效地访问和更新 MapState 。

因此，对流进行排序的更好方法是将乱序流缓存在 MapState<Long,List <Event>> 中，其中 key 是时间戳，而值是均具有相同时间戳的流元素的列表。 这差不多是Flink的SQL和CEP库在需要排序时在内部完成的。
