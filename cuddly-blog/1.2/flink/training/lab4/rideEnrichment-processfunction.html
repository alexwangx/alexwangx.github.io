Lab 4 - 事件时间计时器（到期状态）

这项练习的目的是，以比 以前的练习 更健壮的方式，将每次乘坐的TaxiRide START和TaxiFare记录结合在一起。

在此应用程序中使用 RichCoFlatMap 的问题在于，在实际系统中，我们必须期望某些记录会丢失或损坏。 这意味着随着时间的流逝，我们将积累越来越多的不匹配的 TaxiRide 和 TaxiFare 记录集合，等待与永远不会到达的事件数据进行匹配。 最终，我们的工作将耗尽内存。

您可以通过使用 CoProcessFunction 中可用的计时器来最终解决并清除所有保留的不匹配状态，从而解决此问题。

输入数据

在本练习中，您将使用两个数据流，一个流使用 TaxiRideSource 生成的 TaxiRide 事件，另一个使用 TaxiFareSource 生成的 TaxiFare 事件。

模拟缺失数据

我们已经安排了输入记录中可预测的部分丢失，因此您可以验证自己是否在正确地处理清除相应的状态。

练习代码在 TaxiRides 的 FilterFunction 中执行此操作。它会丢弃所有END事件，以及第1000个START事件。

Java

DataStream<TaxiRide> rides = env
  .addSource(new TaxiRideSource(ridesFile, servingSpeedFactor))
  .filter((TaxiRide ride) -> (ride.isStart && (ride.rideId % 1000 != 0)))

Scala

val rides = env
  .addSource(new TaxiRideSource(ridesFile, servingSpeedFactor))
  .filter { ride => ride.isStart && (ride.rideId % 1000 != 0) }

预期产量

该练习的结果是 Tuple2<TaxiRide，TaxiFare> 记录的数据流，每个不同的 rideId 都有一个。您应将每次乘车起步的事件及其相应的票价数据加入该事件。

为了清楚地了解正在发生的情况，请创建侧面输出，在其中收集在 CoProcessFunction 的 OnTimer 方法中丢弃的每个不匹配的 TaxiRide 和 TaxiFare 。

联接基本正常后，就不用再打印联接记录了。 相反，请打印输出到侧面输出的所有数据，并验证结果是否合理。 如果使用上面建议的过滤器，则应该会看到类似的内容。 这些是 TaxiFare 记录，曾在状态中存储了一段时间，但由于匹配的 TaxiRide 事件尚未到达而最终被丢弃。
