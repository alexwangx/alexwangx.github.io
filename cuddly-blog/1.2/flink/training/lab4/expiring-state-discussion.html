Lab 4 - Discussion

此练习的参考解决方案说明了一种思考如何管理否则可能泄漏的状态的方式，但这并不是只有一个明显正确的答案的情况。 该练习的目的是激发人们对如何使用状态和计时器的思考，并将其中涉及的一些问题浮出水面。

一个好的解决方案的目标是什么？这应该

  • 产生正确的结果

  • 不泄漏状态

  • 容易理解

  • 性能不错

现在，我们考虑一下这些目标来研究建议的解决方案。我们在 processElement1 中找到了这段代码（顺便说一下， processElement2 是相同的，只是乘车和票价之间的角色相反）：

public void processElement1(TaxiRide ride, Context context, Collector<Tuple2<TaxiRide, TaxiFare>> out) throws Exception {
    TaxiFare fare = fareState.value();
    if (fare != null) {
        fareState.clear();
        context.timerService().deleteEventTimeTimer(fare.getEventTime());
        out.collect(new Tuple2(ride, fare));
    } else {
        rideState.update(ride);
        // as soon as the watermark arrives, we can stop waiting for the corresponding fare
        context.timerService().registerEventTimeTimer(ride.getEventTime());
    }
}

这意味着

  • 每当到达一个不完整的事件时，我们会将其存储在状态中并创建一个计时器

  • 每当到达完成一对配对的事件时，我们都会清除状态并删除匹配事件的计时器（之前已存储）

因此很明显，如果两个事件都到达，则不会泄漏任何内容。但是，如果失踪了怎么办

在这种情况下，计时器将在某个时间触发，并运行以下代码，这将清除所有可能存在的状态：

public void onTimer(long timestamp, OnTimerContext ctx, Collector<Tuple2<TaxiRide, TaxiFare>> out) throws Exception {
    if (fareState.value() != null) {
        ctx.output(unmatchedFares, fareState.value());
        fareState.clear();
    }
    if (rideState.value() != null) {
        ctx.output(unmatchedRides, rideState.value());
        rideState.clear();
    }
}

好的，但是我们如何决定等待多长时间？等到 ride.getEventTime() 够了吗？

为 ride.getEventTime 设置事件时间计时器的作用是等待，直到解决乘坐和票价流中的任何乱序为止。 假设水印是完美的 ，则所有更早的乘车和票价事件都会在水印到达 ride.getEventTime 时到达。

在这些练习中，加水印实际上是完美的-不可能有迟到的事件。 但是在现实环境中，您应该期待一些后期事件，并且我们应该期望我们的实现在这种情况下能够正确运行。 此参考解决方案将执行以下操作：

  • 匹配对中的一个事件将首先到达，并创建一个计时器以安排其最终删除

  • 该计时器将触发，事件将被清除

  • 匹配事件迟到，并创建另一个计时器，在这种情况下，该计时器的时间已经过去

  • 下一个到达的水印触发该计时器，并且状态被清除

换句话说，当事件迟到时，不会泄漏任何状态，但是不会生成结果联接。 因此，如果您希望尽管数据迟到而仍然无法获得结果，则应创建计时器，通过将必要的状态保留一些额外的时间来适应某些延迟，例如，

context.timerService().registerEventTimeTimer(ride.getEventTime() + ALLOWED_LATENESS);

尝试容纳任意延迟事件不是一个好主意，因为这样做需要无限期地为每个延迟事件保持某种状态。
