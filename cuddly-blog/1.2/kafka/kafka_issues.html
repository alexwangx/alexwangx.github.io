Kafka Issues

Offset 丢失的现象

现象生产环境下, kafka consumer出现大量堆积, 消费者会从头消费.

  1. 当前consumer消费的topic的leader发生切换, 切换后的 leader 保存的 offset 与之前的 leader 保存的 offset 状态不一致, 此时 consumer 进行消费时发现 offset 非法.

  2. 会对非法的 partition 进行 offset 重置. 重置规则是根据用户配置的 auto.offset.reset 参数进行重置.

  3. auto.offset.reset 参数可选值为: earliest,latest,none

  4. 当前生产环境配置的是 earliest 参数. 所以再对 offset 进行重置的时候, 采用的是 earliest 策略, 会从头进行消费.

  5. 问题的原因

    • 问题的本质是进行 leader 切换的时候,新 leader 与旧 leader 的 offset 状态不一致,才会导致这个问题.

    • 不一致的原因是: 从 kafka 0.11版本, kafka 默认允许脏 leader 竞选. (unclean.leader.election.enable=true 默认值为 true) , 当产生脏 leader 竞选时, 就有可能会出现新旧 leader 的状态不一致.

  6. ISR

    • isr Controller是维护 leader 跟 flower (有效副本)的一个列表集合.

    • isr 剔除规则. (maxLagMs默认10秒)

      • 由于网络,机器负载等等一些原因, 这个replica fetch的进程卡住了,该replica的leo超过maxLagMs没有更新了。

      • follower的拷贝速度远远跟不上leader的写入速度,已经持续落后了maxLagMs.

        def getOutOfSyncReplicas(leaderReplica: Replica, maxLagMs: Long): Set[Replica] = {
    /**
     * there are two cases that will be handled here -
     * 1. Stuck followers: If the leo of the replica hasn't been updated for maxLagMs ms,
     *                     the follower is stuck and should be removed from the ISR
     * 2. Slow followers: If the replica has not read up to the leo within the last maxLagMs ms,
     *                    then the follower is lagging and should be removed from the ISR
     * Both these cases are handled by checking the lastCaughtUpTimeMs which represents
     * the last time when the replica was fully caught up. If either of the above conditions
     * is violated, that replica is considered to be out of sync
     *
     **/
    val candidateReplicas = inSyncReplicas - leaderReplica

    val laggingReplicas = candidateReplicas.filter(r => (time.milliseconds - r.lastCaughtUpTimeMs) > maxLagMs)
    if (laggingReplicas.nonEmpty)
      debug("Lagging replicas are %s".format(laggingReplicas.map(_.brokerId).mkString(",")))

    laggingReplicas
  }

  7. Leader 竞选

    • [topic1,10], ISR: [1,2], leader:2

    • replica 1因为跟不上 leader 或者超时, 从 ISR 中被踢出来, ISR: [2]

    • leader 2 当前机器由于网络或者负载高等原因, 导致请求 zk 超时, zk 上面的 leader 目录被删除从而触发 unclean 选举.选举了一个不在 ISR 中的副本作为 leader 3.

    • consumer 继续从上一次消费完的 offset poll数据, 但是这个 offset 在 leader3的 log 中不存在, 进而触发auto.offset.reset, 这个因为设置了 earliest, 因此从 replica 3的 log 保存的最早的一条数据开始消费. 造成了 offset 丢失现象.

    • 当 broker 2重新建立会话链接后, 发现 leader 已经不是自己了, 成为了 follower, 此时会对 log 进行截断.保持跟 leader 的状态一直.

      private boolean updateFetchPositions() {
        if (subscriptions.hasAllFetchPositions())
            return true;

        // If there are any partitions which do not have a valid position and are not
        // awaiting reset, then we need to fetch committed offsets. We will only do a
        // coordinator lookup if there are partitions which have missing positions, so
        // a consumer with manually assigned partitions can avoid a coordinator dependence
        // by always ensuring that assigned partitions have an initial position.
        coordinator.refreshCommittedOffsetsIfNeeded();

        // If there are partitions still needing a position and a reset policy is defined,
        // request reset using the default policy. If no reset strategy is defined and there
        // are partitions with a missing position, then we will raise an exception.
        subscriptions.resetMissingPositions();

        // Finally send an asynchronous request to lookup and update the positions of any
        // partitions which are awaiting reset.
        fetcher.resetOffsetsIfNeeded();

        return false;
    }

public void resetMissingPositions() {
        final Set<TopicPartition> partitionsWithNoOffsets = new HashSet<>();
        for (PartitionStates.PartitionState<TopicPartitionState> state : assignment.partitionStates()) {
            TopicPartition tp = state.topicPartition();
            TopicPartitionState partitionState = state.value();
            if (partitionState.isMissingPosition()) {
                if (defaultResetStrategy == OffsetResetStrategy.NONE)
                    partitionsWithNoOffsets.add(tp);
                else
                    partitionState.reset(defaultResetStrategy);
            }
        }

        if (!partitionsWithNoOffsets.isEmpty())
            throw new NoOffsetForPartitionException(partitionsWithNoOffsets);
    }

  8. 解决方案

    • topic 副本数保证为3副本.

    • ack 机制设置为acks=-1

    • 禁用藏 leader 竞选. unclean.leader.election.enable=false.

    • 调大一些请求超时时间.

ZK Session

Fix Version/s:3.4.6

  1. 即zk的session过期和ephemeral node删除并不是一个原子操作, 所以带来的问题的过程如下,

    • client session超时, 它会假设在zk, 这个ephemeral node已经被删除, 但是zk当前由于某种原因hang住了, 比如very long fsync operations, 所以其实这个ephemeral node并没有被删除

    • 但client是认为ephemeral node已经被删除, 所以它会尝试重新创建这个ephemeral node, 但得到的结果是NodeExists, 因为这个node并没有被删除, 那么client想既然有就算了

    • 这个时候, zk从very long fsync operations的hang中恢复, 它会继续之前没有完成的操作, 把ephemeral node删掉

    • 这样client就会发现ephemeral node不存在了, 虽然session并没有超时
