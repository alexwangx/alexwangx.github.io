意外删除

hadoop fs -rm -r -skipTrash /

如果误操作执行了上面命令, ctrl-c 的命令不会停止删除! 立刻!!! 马上!!! 停止 NameNode, DataNode 服务. 并备份NN 元数据!!!

如何尽量挽回

备份元数据信息, 2 个 namenode, 3 个 journalnode 元数据(可不备份 journalnode).

  1. active nn 上面的操作.

    • cd /path_your_nn_data/current # 切换到元数据目录

    • 查看 edits 及 image 文件,搜索 OP_DELETE 确认你执行的删除目录所在的 edits 文件.

      hdfs oev -i edits_0000000002859034507-0000000002859037793 -o edits.xml

hdfs oiv -i fsimage_0000000002859037793 -o fsimage.ls

    • 删除找到的 edit 文件或者 image 文件, 及该文件修改时间之后的文件.

      rm -rf  edits_0000000002859034507-0000000002859037793
rm -rf  edits_0000000002859037794-0000000002859038006
rm -rf  edits_0000000002859038007-0000000002859038174
rm -rf  edits_0000000002859038175-0000000002859038363
rm -rf  edits_0000000002859038364-0000000002859038542
rm -rf  edits_0000000002859038543-0000000002859038785
rm -rf  edits_inprogress_0000000002859038786

    • 修改 seen_txid 文件内的 id 为删除后最后的edits 文件的 sessionid

  2. journalnode

    • cd /path_your_jn_data/current # 切换到元数据目录

    • 保留 version 文件, 删除其他文件(三个节点都执行该操作)

  3. 服务启动

    • 启动三台 journalnode 服务.

    • 启动 zkfc 服务.

    • 启动 active nn 服务. (查看是否有异常,无异常启动 DN 服务)

    • 启动 DN 服务.(查看上报数据块是否正常)

      • active NN 会开启安全模式, 如果上报的数据块达不到一定比例,安全模式不会关闭, 可手动 leave 安全模式.

      • active NN 丢失的数据块 就是删除丢失的数据, 想要恢复的话 只能磁盘找回(比较麻烦)

    • 确认HDFS读写功能是否正常.

    • 启动后执行:(不需要操作)

      • hdfs dfsadmin -safemode enter

      • hdfs dfsadmin -saveNamespace

      • 重启 active nn

  4. standby nn 服务

    • hdfs namenode -bootstrapStandby # 清理掉所有standby nn 上面的元数据文件

    • 启动 standby nn

      • 观察是否同步 image 文件

  5. 统计丢失的文件.

    • 已经终止的后面继续删除的文件.
